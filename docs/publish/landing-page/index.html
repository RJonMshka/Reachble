The problem with "in the tree"

Research shows 60–80% of CVEs flagged by dependency scanners sit in code paths that are never called. The vulnerable function lives in your node_modules — but your application never imports it, never invokes it, never reaches it.

With SBOM mandates now in effect (US EO 14028, EU CRA), you need a VEX document — a machine-readable justification for every CVE you're closing as not exploitable. Right now that's a manual spreadsheet. Reachble generates it automatically from your code.

Verdict tiers

Not a severity number. An exploitability verdict grounded in your actual code.

CRITICAL Reachable from unauthenticated external input — HTTP, CLI, file, env variable
HIGH Reachable from authenticated route or internal service call
LOW Reachable in code, but no external input path found
SAFE Vulnerable symbol never reached → VEX not_affected, justification included

How it works

1
Lockfile parsing Reads your package-lock.json, yarn.lock, or pnpm-lock.yaml to build a complete dependency inventory — including depth, devOnly flag, and transitives.
2
CVE resolution Queries OSV.dev, NVD, and GHSA for each package version. Extracts affected symbols (specific functions/classes) from affected_functions fields, fix-commit diffs, and description parsing.
3
Import-graph reachability Uses @typescript-eslint/parser to build a static import graph across your project — no tsconfig required. Maps whether the vulnerable symbol is imported anywhere in your reachable code.
4
VEX output Emits CycloneDX VEX + OpenVEX with machine-checkable evidence for every verdict. SAFE verdicts carry a not_affected justification you can ship with your SBOM.
